# Merchant 取消发货与受控重新发货设计

- 日期：2026-06-05
- 状态：Draft（需求与设计已逐项确认）
- 范围：`backend` + `merchant`

## 1. 背景

当前 Merchant 端对待发货订单支持“发货”，对已发货订单只支持“完成”，没有发货纠错能力。

实际运营中可能出现物流公司或物流单号填写错误、仓库取消出库等情况。管理员需要撤销本地发货状态，让客户重新看到“待发货”，之后使用正确物流信息再次发货。

微信小程序发货信息管理服务不提供撤销发货接口。根据官方 `uploadShippingInfo` 文档：

1. 首次发货与重新发货均调用 `/wxa/sec/order/upload_shipping_info`。
2. 对已完成发货的支付单再次调用该接口，视为重新发货。
3. 修改发货模式或物流模式，也视为重新发货。
4. 每笔支付单仅有一次重新发货机会。
5. 更新物流信息时，新的 `upload_time` 必须晚于上次请求。
6. 重复使用重新发货机会会返回 `10060003`。
7. 相关官方文档最近一次变更记录为 2026-05-09。

因此，本功能不能设计成无限次编辑物流，而应提供一次受控的“取消发货 -> 再次发货”纠错闭环。

## 2. 目标与非目标

### 2.1 目标

1. Merchant 支持对符合条件的已发货订单执行一次“取消发货”。
2. 取消后订单恢复为 `paid`，客户看到“待发货”，不再展示已作废的当前物流信息。
3. 保存取消前的完整物流快照、操作人、原因和微信同步情况，供管理员审计。
4. 取消后复用现有“发货”入口完成重新发货，不新增独立“重新发货”入口。
5. 微信支付订单重新发货时继续调用现有微信发货接口，并遵守微信唯一一次重新发货限制。
6. 所有支付方式使用一致的本地规则：每个订单只允许取消发货一次。
7. 排除所有海尔订单，避免与易理货/海尔履约状态冲突。

### 2.2 非目标

1. 不提供任意次数的物流编辑。
2. 不新增单独的“重新发货”按钮或页面。
3. 不尝试撤销微信侧已经录入的旧发货信息。
4. 不支持海尔订单、海尔子订单的取消发货。
5. 不修改海尔/易理货接口。
6. 取消发货时不向客户发送站内通知或微信订阅消息。

## 3. 方案对比与决策

### 3.1 方案 A：独立发货操作记录

新增 `OrderShippingAction`，记录首次发货、取消发货和重新发货。

优点：

1. 保存完整历史，满足审计与故障排查。
2. 可以可靠判断订单是否已使用取消发货机会。
3. 不把快照和操作历史继续堆积在 `Order` 模型。
4. 后续如需展示发货历史，可直接复用。

缺点：需要新增模型和数据库迁移。

### 3.2 方案 B：字段直接放在订单上

给 `Order` 增加取消次数、旧物流快照、待重新发货等字段。

优点：实现较快。

缺点：只能保存有限历史，订单模型职责继续膨胀，审计能力不足。

### 3.3 方案 C：完整物流版本模型

每次发货创建物流版本，订单只指向当前版本。

优点：适合复杂多版本物流管理。

缺点：明显超出本次“一次纠错”的需求，改动和迁移成本过高。

### 3.4 决策

采用方案 A。它覆盖当前需求，并为管理员保留必要的操作证据，但不引入完整物流版本系统。

## 4. 业务规则

### 4.1 允许取消发货

订单必须同时满足：

1. 当前状态为 `shipped`。
2. 不是海尔订单。
3. 从未成功执行过 `cancel_shipping`。

支付方式不作为限制条件。微信支付、信用支付及其他本地支持的支付方式均可使用。

### 4.2 海尔订单识别

不能只判断 `order_type == "haier"`，需要兼容历史订单。满足任一条件即视为海尔订单：

1. `order_type == "haier"`。
2. 存在 `haier_so_id`、`haier_order_no` 或有效 `haier_status`。
3. 任一订单商品的 `product.source == "haier"`。
4. 兼容旧单商品订单时，`order.product.source == "haier"`。

`order_type == "local"` 的本地子订单允许取消发货，只排除真正包含海尔履约的订单。

### 4.3 次数限制

1. 每个订单最多成功取消发货一次。
2. 取消后可以再次发货。
3. 重新发货成功后永久禁止再次取消发货。
4. 非微信订单同样执行一次限制，保证规则一致并防止反复回退状态。
5. 数据库层使用条件唯一约束限制每个订单只能存在一条成功的 `cancel_shipping` 记录，接口层同时使用行锁防止并发重复操作。

### 4.4 客户侧表现

取消发货成功后：

1. 订单状态从 `shipped` 回到 `paid`。
2. 客户看到“待发货”。
3. 当前物流展示字段被清空。
4. 不创建客户通知。
5. 微信侧旧发货记录保持不变，系统不声称已撤销微信发货。

## 5. 数据模型

新增 `OrderShippingAction`：

| 字段 | 说明 |
| --- | --- |
| `order` | 关联订单 |
| `action` | `ship`、`cancel_shipping`、`reship` |
| `status` | `succeeded`、`failed` |
| `shipping_snapshot` | 当次完整物流快照 |
| `operator` | 管理员或客服，可为空以兼容系统操作 |
| `reason` | 取消原因或操作备注 |
| `wechat_sync_required` | 本次是否需要同步微信 |
| `wechat_synced` | 微信同步是否成功 |
| `wechat_response` | 微信响应或错误快照，敏感值需脱敏 |
| `created_at` | 操作时间 |

物流快照至少包含：

1. `logistics_no`
2. `shipping_info`
3. `delivery_record_code`
4. `sn_code`
5. `delivery_images`

管理员历史记录必须能还原取消前物流状态，但这些快照不通过客户订单接口暴露。

## 6. 后端接口设计

### 6.1 取消发货接口

```text
PATCH /api/orders/{id}/cancel_shipping/
```

请求：

```json
{
  "reason": "物流单号填写错误"
}
```

权限：

- 仅管理员或 `support`。

参数约束：

- `reason` 去除首尾空白后必须为 1 至 200 个字符。

事务流程：

1. 使用 `select_for_update()` 锁定订单。
2. 校验原因非空。
3. 校验订单状态为 `shipped`。
4. 使用统一海尔识别逻辑排除海尔订单。
5. 校验不存在成功的 `cancel_shipping` 记录。
6. 保存当前物流完整快照。
7. 创建成功的 `cancel_shipping` 操作记录。
8. 清空当前物流展示字段。
9. 通过状态机将订单从 `shipped` 转回 `paid`。
10. 写入 `OrderStatusHistory`，备注包含取消原因，但不重复写入完整敏感快照。
11. 不调用微信接口，不创建客户通知。

建议清空：

- `logistics_no`
- `shipping_info`
- `delivery_record_code`
- `sn_code`
- `delivery_images`

### 6.2 状态机

新增受控状态转换：

```text
shipped -> paid
```

该转换只由取消发货业务入口使用。接口负责附加海尔、次数、原因和权限限制，不能提供通用状态编辑入口。

`paid`、`shipped` 均计入销量，因此该转换不会改变当前销量统计。

### 6.3 首次发货记录

现有 `/ship/` 成功后创建操作记录：

1. 普通首次发货：`action=ship`。
2. 存在成功的 `cancel_shipping` 且尚无成功 `reship`：`action=reship`。

旧订单没有 `ship` 操作记录时，取消接口直接以当前订单物流字段生成快照，不要求执行历史数据回填。

### 6.4 重新发货

取消后的订单继续使用现有：

```text
PATCH /api/orders/{id}/ship/
```

识别规则：

- 存在成功的 `cancel_shipping`，且不存在成功的 `reship`，则本次为重新发货。

微信支付订单：

1. 继续调用 `/wxa/sec/order/upload_shipping_info`。
2. 使用当前请求生成的 RFC 3339 `upload_time`，确保晚于上次请求。
3. 微信成功后创建 `reship` 记录并将订单转为 `shipped`。
4. 微信返回 `10060003` 时，向 Merchant 返回“该支付单已使用微信重新发货机会”。
5. 微信同步失败时事务回滚，订单保持 `paid`，当前物流信息不保存。
6. 主事务回滚后，以独立短事务写入 `status=failed` 的 `reship` 操作记录，保存本次尝试的物流快照和脱敏微信响应，便于管理员核对；失败记录不视为已经完成本地重新发货。

非微信订单或无需微信同步的订单：

1. 只完成本地重新发货。
2. 创建成功的 `reship` 记录。
3. 订单转为 `shipped`。

### 6.5 订单能力字段

订单响应增加非敏感能力字段：

- `can_cancel_shipping`
- `is_reshipment_pending`
- `reship_requires_wechat_sync`
- `shipping_cancel_count`

这些字段用于 Merchant 控制按钮和提示，Merchant 不自行根据支付方式或运行配置推断是否需要微信同步。字段不包含历史物流快照。

### 6.6 发货操作记录接口

为避免客户接口暴露历史物流信息，增加管理员专用接口：

```text
GET /api/orders/{id}/shipping_actions/
```

权限：

- 仅管理员或 `support`。

按时间倒序返回发货、取消发货和重新发货记录及物流快照。

## 7. Merchant 设计

### 7.1 操作入口

已发货订单满足 `can_cancel_shipping=true` 时，在操作列显示：

```text
取消发货
```

以下情况不显示：

1. 海尔订单。
2. 非 `shipped` 状态。
3. 已执行过取消发货。
4. 已完成、退货中、退款中或已退款。

### 7.2 取消发货弹窗

采用一个弹窗完成原因输入和最终确认，不连续弹两次。

内容：

1. 当前物流公司和物流单号。
2. 必填字段：取消原因。
3. 固定风险提示：

```text
取消后订单将恢复为“待发货”，当前物流信息会被清空。
微信侧原发货记录无法撤销，再次发货将使用唯一一次重新发货机会。
每个订单只能取消发货一次。
```

提交按钮：

```text
确认取消发货
```

### 7.3 取消成功

1. 关闭弹窗并刷新订单列表。
2. 状态显示为“待发货”。
3. 操作列重新出现现有“发货”按钮。
4. 不显示已作废物流信息。

### 7.4 重新发货弹窗

当 `is_reshipment_pending=true`：

1. 复用现有发货表单。
2. 标题显示“重新发货”。
3. 微信支付且需要同步微信时显示：

```text
本次将更新微信发货信息，并使用该支付单唯一一次重新发货机会，提交后不可再次修改。
```

非微信订单不显示微信机会提示，但仍受本地一次取消限制。

### 7.5 发货操作记录

Merchant 订单详情新增“发货操作记录”区域：

1. 展示动作、结果、操作人、原因、时间。
2. 展示管理员可见的物流快照。
3. 展示微信是否要求同步、是否成功及脱敏后的响应摘要。
4. 客户端不展示该区域。

## 8. 错误处理

后端返回明确业务错误：

| 场景 | 错误信息 |
| --- | --- |
| 原因为空 | `取消原因不能为空` |
| 非已发货状态 | `仅已发货订单可以取消发货` |
| 海尔订单 | `海尔订单不支持取消发货` |
| 已取消过 | `该订单已使用取消发货机会` |
| 微信重发机会已使用 | `该支付单已使用微信重新发货机会` |
| 微信同步失败 | 使用现有微信错误映射，并保持本地待发货 |

如果微信请求结果不明确，例如请求超时但可能已被微信接收：

1. 不自动假定重新发货成功。
2. 保持本地 `paid`。
3. 记录脱敏错误日志。
4. 管理员应先通过微信订单查询或微信后台核对，再决定是否重试，避免误耗重发机会。

## 9. 并发与一致性

1. 取消发货使用订单行锁。
2. 条件唯一约束保证每单只能成功取消一次。
3. 发货接口继续在事务中完成本地状态和微信同步。
4. 微信重发失败时，本地物流和状态整体回滚。
5. `OrderShippingAction` 与订单状态变化在同一事务内写入。
6. Merchant 提交期间禁用按钮，防止前端重复请求。

## 10. 测试与验收

### 10.1 后端测试

1. 普通微信订单取消后从 `shipped` 恢复为 `paid`。
2. 信用支付订单可以取消并重新发货。
3. `order_type=local` 的本地子订单允许取消。
4. `order_type=haier` 的海尔子订单拒绝取消。
5. 通过海尔字段或商品来源识别的旧海尔订单拒绝取消。
6. 非 `shipped` 状态拒绝取消。
7. 原因为空时拒绝取消。
8. 取消时保存完整物流快照。
9. 取消后当前物流字段被清空。
10. 取消后创建 `shipped -> paid` 状态历史。
11. 取消时不创建客户通知。
12. 同一订单第二次取消被拒绝。
13. 两个管理员并发取消时只有一个成功。
14. 取消后的发货被识别为 `reship`。
15. 微信重发成功后记录 `reship` 并转为 `shipped`。
16. 微信返回 `10060003` 时返回明确错误并保持 `paid`。
17. 微信重发失败时物流字段、状态和成功操作记录全部回滚。
18. 非微信订单重新发货只执行本地流程。
19. 旧订单没有 `ship` 记录时仍能从当前物流字段生成取消快照。

### 10.2 Merchant 测试

1. 仅符合条件的订单显示“取消发货”。
2. 海尔订单不显示按钮。
3. 弹窗展示当前物流信息和固定风险提示。
4. 未填写原因不能提交。
5. 取消成功后刷新为“待发货”。
6. `is_reshipment_pending=true` 时发货弹窗显示“重新发货”。
7. 微信订单显示唯一重发机会提示。
8. 操作记录只在 Merchant 详情中展示。
9. 请求进行中按钮不可重复点击。

### 10.3 回归测试

1. 普通首次发货流程保持不变。
2. 海尔推送和物流查询流程保持不变。
3. 完成订单、确认收货、退货和退款状态流转保持不变。
4. 客户端订单列表和详情只读取当前状态与当前物流，不暴露历史快照。

## 11. 发布与回滚

### 11.1 发布

1. 先执行数据库迁移。
2. 发布后端接口和能力字段。
3. 发布 Merchant 操作入口和详情记录。
4. 使用非海尔测试订单验证取消与重新发货。
5. 微信测试需使用可消耗重新发货机会的测试支付单，避免在真实订单上重复试验。

### 11.2 回滚

1. Merchant 可先回滚按钮和入口，阻止新操作。
2. 后端接口可回滚，但保留操作记录表和已产生的审计数据。
3. 已取消发货的订单不能通过代码回滚自动恢复旧物流，应由管理员根据快照人工重新发货。

## 12. 结论

本方案只新增一次受控的取消发货能力，并复用现有发货流程完成重新发货。它避免开放无限物流编辑，遵守微信每笔支付单仅一次重新发货的限制，同时排除缺少对应接口能力的海尔订单，保证本地状态、微信同步和管理员审计之间的边界清晰。
