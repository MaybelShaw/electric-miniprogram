# 小程序未登录购买改为跳转登录并回跳恢复流程（方案2）设计

- 日期：2026-04-28
- 状态：Draft（已确认采用方案2）
- 范围：`frontend`（Taro 小程序）

## 1. 背景

当前前端在多个未登录场景仅提示“请先登录”，不会直接进入登录授权流程。对“立即购买/加入购物车”等强目标操作，用户需要额外手动进入登录，链路中断，转化受损。

本次目标是将关键未登录场景改为统一登录守卫：未登录即跳登录页；登录成功后回到触发页面并继续当前流程。

## 2. 目标与非目标

### 2.1 目标

1. 未登录触发关键受限操作时自动跳转登录授权页。
2. 登录成功后回到触发页面。
3. 对商品详情页购买相关操作支持“恢复当前流程”（buy/cart intent 恢复）。
4. 统一登录守卫逻辑，减少页面级重复实现。
5. 新建独立登录页承载授权与回跳，不复用个人中心页面。

### 2.2 非目标

1. 不改动后端鉴权与 token 刷新机制。
2. 不引入全局路由拦截器。
3. 不在本次统一重构所有“请先登录”提示页面（仅覆盖已确认范围）。

## 3. 方案对比与决策

### 3.1 候选方案

1. 方案1：各页面就地改 toast 为跳转。
2. 方案2：新增统一守卫工具（本次采用）。
3. 方案3：全局路由层拦截。

### 3.2 选择方案2原因

1. 支持回跳 + intent 恢复的复杂链路，复用价值高。
2. 改动面可控，不需要一次性重构路由层。
3. 能避免方案1在多页面复制同类逻辑导致行为不一致。
4. 独立登录页能避免个人中心页面职责混杂和副作用。

## 4. 详细设计

## 4.1 登录守卫工具

新增前端工具函数（命名待落地时按项目习惯定，如 `requireLogin`）：

- 输入参数：
  - `intent`：可选，当前动作意图（如 `buy`、`cart`）
  - `redirect`：可选，默认自动取当前页面完整路径
- 行为：
  1. 若已登录（存在有效 token），返回 `true`
  2. 若未登录，跳转 `/pages/login/index`，并携带：
     - `redirect`：当前页路径
     - `intent`：动作意图（可选）
     返回 `false`

约束：

- 跳转参数必须统一编码/解码，防止 query 串解析错误。
- 仅封装“是否需要登录 + 跳转”，不承载业务动作实现。

## 4.2 独立登录页与回跳协议

新增独立页面：`pages/login/index`（并在 `app.config.ts` 注册）。

登录页读取 query：

- `redirect`：回跳页面
- `intent`：可选，透传给目标页面

登录成功后：

1. 有 `redirect`：`redirectTo` 到 `redirect`（附带 `intent`）
2. 无 `redirect`：跳默认页（建议个人中心）

失败处理：

- 用户取消授权/登录失败：停留登录页提示，不做循环跳转。

## 4.3 页面接入范围（本次）

### A. 商品详情页（重点）

文件：`frontend/src/pages/product-detail/index.tsx`

- 未登录点击“立即购买/加入购物车”时调用守卫并传 `intent=buy|cart`。
- 回跳后页面检测 `intent` 且用户已登录，执行一次自动恢复：
  - `buy`：恢复购买流程（例如弹数量弹层）
  - `cart`：恢复加购流程
- 使用一次性标记（如 `hasAutoResumed`）避免重复恢复。

### B. 订单详情页

文件：`frontend/src/pages/order-detail/index.tsx`

- 将受限操作入口改为守卫触发，未登录直接去登录页。
- 回跳后仅恢复页面上下文，不要求自动执行风险动作（默认用户再点一次）。

### C. 消息中心页

文件：`frontend/src/pages/message-center/index.tsx`

- 进入页面即守卫，未登录直接跳登录。

### D. 企业认证页

文件：`frontend/src/pages/company-certification/index.tsx`

- 页面进入守卫，未登录直接跳登录。

## 4.4 暂不改动范围

1. 购物车未登录空态（已有“立即登录”按钮，先保留）
2. 个人中心浏览态入口（保留当前弱引导）

## 5. 数据流与状态流

1. 用户在 A 页触发受限动作 -> `requireLogin(intent)`
2. 未登录 -> 跳独立登录页（携带 redirect + intent）
3. 登录成功 -> 跳回 A 页（透传 intent）
4. A 页检测到 intent + 已登录 -> 执行一次恢复动作 -> 清理/失效 intent

## 6. 风险与处理

1. 风险：intent 重复执行（返回后多次 onShow）
   - 处理：一次性恢复标记 + 执行后立即清理。
2. 风险：redirect 非法或解析失败
   - 处理：fallback 到默认页，不抛异常阻断。
3. 风险：不同页面对自动恢复语义不一致
   - 处理：本次仅在商品详情启用自动恢复，其余页面仅回跳不自动执行敏感动作。
4. 风险：新增登录页后与旧跳转路径冲突
   - 处理：统一替换到 `/pages/login/index`，并在 app 配置中注册页面。

## 7. 测试与验收

### 7.1 功能验收

1. 未登录商品详情点“立即购买” -> 跳登录 -> 登录后回原商品并自动恢复购买流程。
2. 未登录商品详情点“加入购物车” -> 跳登录 -> 登录后回原商品并自动恢复加购流程。
3. 未登录进消息中心/企业认证页 -> 跳登录 -> 登录后回原页。
4. 订单详情受限入口未登录 -> 跳登录 -> 登录后回原页。
5. redirect 缺失时走默认兜底页。

### 7.2 回归检查

1. 已登录用户路径无变化。
2. 401 全局拦截（`utils/request.ts`）不与新守卫冲突。
3. 不出现登录页与来源页互跳循环。
4. 新增登录页路由可访问且未破坏 TabBar 导航。

## 8. 发布与回滚

### 8.1 发布

1. 合并前端改动。
2. 在小程序开发者工具做登录/回跳/恢复动作冒烟。
3. 重点观察商品详情页未登录购买链路。

### 8.2 回滚

- 仅需回退本次前端守卫接入与新增登录页改动；后端无需联动回滚。

## 9. 实施清单（供后续计划拆分）

1. 新增登录守卫工具（参数协议：redirect + intent）。
2. 新建 `pages/login/index` 独立登录页并注册路由。
3. 登录页实现回跳与 intent 透传。
4. 商品详情页接入守卫并实现一次性自动恢复。
5. 订单详情页、消息中心、企业认证页接入守卫。
6. 冒烟并补充必要前端单测（若项目现有测试基建允许）。

## 10. 结论

采用方案2可在可控改动范围内实现“未登录自动跳登录 + 登录后回跳 + 关键流程恢复”，并通过独立登录页确保职责清晰、减少对个人中心页面的副作用。
