# 登录页视觉对齐 Profile 风格（方案A）设计

- 日期：2026-04-28
- 状态：Draft（已确认）
- 范围：`frontend/src/pages/login/*`

## 1. 背景

新建独立登录页已满足“未登录跳转 + 登录回跳”功能，但当前视觉样式与小程序现有主风格（尤其 `profile` 页）不一致，影响整体体验统一性。

## 2. 目标与非目标

### 2.1 目标

1. 登录页视觉风格与 `profile` 页核心卡片区一致。
2. 保留现有登录业务逻辑，不改变回跳和 intent 机制。
3. 统一使用现有样式变量体系，减少硬编码。

### 2.2 非目标

1. 不重构 `profile` 页。
2. 不新增授权流程步骤（协议弹窗、二次确认等）。
3. 不抽取跨页面公共样式（本次先快速落地方案A）。

## 3. 方案选择

采用方案A：直接在登录页复用 `profile` 的视觉语义（布局、卡片语言、按钮层级），不做抽象层改造。

### 3.1 选择原因

1. 改动小，交付快。
2. 风格一致性高，效果可预期。
3. 对现有登录功能零侵入。

## 4. 详细设计

## 4.1 页面结构

登录页结构调整为：

1. 顶层容器：`min-height: 100vh`，背景用 `$background-color-base`。
2. 中央卡片：白底、圆角、轻阴影、装饰渐变背景（参考 `profile .user-section`）。
3. 内容层级：
   - 顶部头像/品牌占位（可用默认头像或抽象图形）
   - 主标题“登录授权”
   - 说明文案（弱化色）
   - 主操作按钮“微信授权登录”

## 4.2 视觉规范对齐点

1. 颜色：主色使用 `$primary-color`，背景/文字/边框使用现有变量。
2. 间距：使用 `$spacing-*` 体系。
3. 字号：使用 `$font-size-*` 体系。
4. 圆角与阴影：使用 `$border-radius-*` 和 `$shadow-*`，与 `profile` 卡片观感一致。
5. 按钮：高度、圆角、激活态与现有页面主按钮语言统一。

## 4.3 行为与状态

行为保持现有逻辑不变：

1. 点击登录 -> 调用 `authService.login()`。
2. 登录中：按钮 `loading/disabled`，文案“登录中...”。
3. 登录成功：token 存储 + 事件触发 + 回跳（redirect/intent）。
4. 登录失败：toast 提示，停留当前页。

## 4.4 兼容性与适配

1. 适配常见手机宽度，避免按钮溢出和文字换行异常。
2. 支持安全区底部留白。
3. 不依赖新增第三方组件。

## 5. 风险与应对

1. 风险：登录页与 profile 过度耦合（仅视觉层面）
   - 应对：本次只复用风格语义，不共享选择器；后续如需长期统一再抽公共样式。
2. 风险：改样式引发已有交互回归
   - 应对：不改登录逻辑函数，仅改样式与结构层。

## 6. 验收标准

1. 登录页视觉与 `profile` 用户区域同风格（白卡 + 阴影 + 浅色背景 + 主色按钮）。
2. 登录回跳、intent 透传功能无回归。
3. 小程序开发者工具下主流机型显示正常。
4. 登录页相关文件 lint 通过（若有历史遗留错误，不应由本次引入）。

## 7. 实施清单（供后续计划拆分）

1. 调整 `frontend/src/pages/login/index.tsx` 结构层级。
2. 重写 `frontend/src/pages/login/index.scss`，对齐 `profile` 风格。
3. 运行前端检查并做手工冒烟（登录成功/失败/回跳）。

## 8. 结论

采用方案A可在最小改动下将登录页快速拉齐至小程序现有视觉体系，确保功能稳定且观感统一。
