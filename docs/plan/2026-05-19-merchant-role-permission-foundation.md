# 商户角色权限基础收敛

## 背景

在订单修改和资金分流继续推进前，需要先确认商户管理端的角色边界足够清晰，避免店铺管理员、店铺子管理员、店铺运营在订单、退款、发票、账务等流程中出现越权或误拦截。

本轮先做“集中配置的权限码”基础层，不直接上数据库 RBAC。原因是当前系统已经有 `StoreMember.role`、店铺范围过滤和商户端路由过滤，最小风险方案是先把散落的 `is_staff` 判断收束到统一权限码，再为后续可配置权限表预留形状。

## 角色与能力

- 平台管理员：全平台店铺、用户、店铺成员、订单、账务、支付/结算配置。
- 店铺管理员：本店运营、商品、内容、订单、退款、发票、账务查看、本店成员管理。
- 店铺子管理员：本店运营、商品、内容、订单发货/取消/完成、退货退款、发票、折扣；默认不能改价、不能看账务。
- 店铺运营：本店基础运营、商品/内容、订单查看和发货、发票处理。

## 本轮调整

- 后端新增集中权限码与角色预设：`stores.permissions.ROLE_PERMISSION_PRESETS`。
- `StoreMember` 新增 `store_sub_admin` 角色。
- `/api/stores/current/` 和登录返回的 `store_roles` 暴露成员权限码。
- 订单改价、取消、发货、完成、退货审批、退款处理改为基于店铺权限码判断。
- 店铺管理员可以管理本店子管理员/运营，但不能授予平台管理员或店铺管理员角色。
- 商户端菜单从固定“店铺路由白名单”改为权限码过滤。
- 店铺成员页支持“店铺子管理员”，用户候选改走成员管理端点，避免店铺管理员依赖平台用户列表权限。

## 后续在订单修改/资金分流前还需要确认

- 订单改价：是否允许店铺子管理员改价。如果允许，只需把 `orders.adjust_amount` 加进 `store_sub_admin` 预设。
- 资金分流：店铺支付配置、结算规则、分账比例必须继续只对平台管理员开放，店铺管理员最多查看本店结算结果。
- 订单拆分：主订单、子订单、店铺订单的状态联动需要继续用角色测试覆盖，尤其是退款、取消、部分发货。
- 小程序端：买家只能看自己的订单；店铺范围参数不能影响买家订单查询。

## 验证

- `npm run build` in `merchant/`

## 完成标准

- 角色权限集中配置，不再依赖页面硬编码推断能力。
- 店铺管理员和店铺子管理员的订单操作边界有测试覆盖。
- 商户端菜单按后端返回的权限码展示。
