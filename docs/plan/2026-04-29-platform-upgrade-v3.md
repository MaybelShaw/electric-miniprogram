# 平台升级实施计划 v3

## 概要

本计划用于把当前项目从“单店商城”升级为“平台型店铺服务商系统”，并在不废掉现有主店业务的前提下，分阶段完成三条主线改造：

- 交易主线：支持多子店铺，用户统一购物车、统一支付，支付成功后按“店铺 + 商品 SPU”自动拆子单，后续接微信服务商分账。
- 用户主线：小程序取消当前自动登录模式，改成显式微信快捷登录，首次登录即完成手机号授权；昵称默认生成，并支持登录后补充头像昵称。
- 体验主线：小程序建立新的设计系统，并完成全站现代化 UI 改版，核心风格为“现代生活方式”。

本版在原 v2 方案基础上，补充一个新的首页“活动/优惠专区板块”，并明确该板块复用现有专区机制，而不是单独再造一套活动系统。

## 当前基础

当前系统已经具备专区能力的基础闭环：

- 后端已有 `SpecialZoneCover` 作为首页专区封面配置。
- 后端已有 `HomeBanner.position` 作为专区页轮播配置。
- 商品已有 `show_in_gift_zone`、`show_in_designer_zone`、`show_in_best_seller_zone` 控制专区商品曝光。
- 商家后台已有 `HomeBanners`、`SpecialZoneCovers`、`Products` 三个配置入口。
- 小程序首页和 `special-zone` 页面已经消费这条链路。

因此新增“活动板块”的实现方式固定为：新增第四种专区类型 `promotion`，沿用现有专区体系完成首页入口、专区页轮播、商品筛选、后台配置和文档沉淀。

## 目标能力

### 1. 多店铺平台化

- 新增平台域 `stores`，承载店铺、成员、页面配置、支付配置、结算规则等能力。
- 当前全部历史业务数据归属 `main_store`。
- 只有 `main_store` 允许启用海尔能力。
- 子店商品、分类、品牌、Banner、案例、专区封面、首页配置都独立维护。
- 平台管理员可管理所有店铺；店铺管理员和员工只能操作本店数据。

### 2. 订单重构为“结算单 + 子单”

- 支持跨店商品统一购物车、统一支付。
- 支付成功后按 `store_id + product_id(SPU)` 自动拆子单。
- 同一店铺同一商品 SPU 的多个 SKU/数量保留在同一张子单里，作为多个订单行。
- 用户端默认查看子单，不以聚合主单作为主要展示对象。
- 后台保留 `CheckoutOrder -> SubOrder[]` 的追踪关系。

### 3. 微信快捷登录

- 取消当前“用户点一下就隐式创建账号”的登录体验。
- 新流程为：点击微信快捷登录 -> `Taro.login()` -> 手机号授权 -> 后端换取身份并创建/复用用户 -> 返回登录态。
- 首次登录手机号必拿。
- 昵称默认自动生成，后续在“我的”页引导补充头像与昵称。
- 商品分享链路纳入同一套登录改造：
  当用户通过分享商品链接首次进入小程序，或当前处于未登录状态时，允许先浏览商品详情，但点击“立即购买”或其他交易动作时必须先跳转登录/注册/授权流程。
- 登录成功后需要自动回跳到原始分享商品页，并继续用户刚才触发的购买动作，不能让用户登录后丢失商品上下文。
- 该场景与普通站内浏览的登录拦截共用一套能力，不单独维护第二套分享页登录逻辑。

### 4. 小程序 UI 重构

- 先建立设计系统，再统一改造核心页面。
- 第一批页面包含：首页、店铺专区页、店铺首页、商品列表、商品详情、购物车、下单、支付结果、订单列表、订单详情、我的页、登录弹层/登录页。
- 风格采用“现代生活方式”，降低传统蓝白商城感，强化内容感、品牌感和运营板块识别度。

## 首页活动/优惠专区板块

### 产品定义

新增一个“优惠专区/活动专区”板块，定位为首页的可配置运营入口，而不是临时活动弹窗或独立促销系统。

行为固定如下：

- 该板块在首页展示为独立专区入口。
- 展示位置放在现有“爆品专区”之后、“品类专区”之前。
- 小程序点击后跳转到现有专区页 `special-zone`，通过 `type=promotion` 和标题参数进入。
- 活动专区页继续沿用当前专区页结构：顶部轮播图、下方商品列表。
- 活动专区商品来自商品维度的专区开关，不从首页活动板块里单独维护商品列表。
- 当后台没有启用活动专区封面时，首页直接隐藏该板块。
- 当活动专区没有商品时，专区页显示空态，不影响首页其他内容。

### 后端扩展

后端不新增全新活动模型，直接扩展现有专区模型和字段：

- `catalog.Product`
  新增 `show_in_promotion_zone: bool`
- `catalog.HomeBanner.position`
  新增 `promotion`
- `catalog.SpecialZoneCover.type`
  新增 `promotion`

同步扩展的后端行为：

- 商品列表接口支持 `show_in_promotion_zone` 查询参数。
- 商品导出、筛选、后台编辑回显同步支持该字段。
- 首页专区图片接口支持 `type=promotion`。
- 首页轮播图接口支持 `position=promotion`。
- `special-zone` 页使用的商品查询逻辑支持 `promotion` 分支。
- 若后续多店改造引入 `store_id`，该活动专区能力默认也按店铺隔离，不做跨店共享。

数据策略：

- 通过一条 `catalog` migration 增加新布尔字段和新 choices。
- 不创建默认 promotion 数据。
- 历史数据不回填；新板块完全由后台运营创建后生效。

### 商家后台配置

商家后台继续沿用现有三个配置入口，不增加新菜单：

- `Products` 页面“专区展示”区域新增一个开关：
  `优惠专区展示 / show_in_promotion_zone`
- `HomeBanners` 页面“展示位置”下拉新增：
  `优惠专区`
- `SpecialZoneCovers` 页面“专区类型”下拉新增：
  `优惠专区`

后台运营配置链路：

1. 上传活动专区封面图
2. 配置活动专区轮播图
3. 在商品编辑页勾选“优惠专区展示”
4. 小程序首页自动出现活动专区入口
5. 用户点击进入活动专区页浏览被勾选的商品

后台不新增独立“活动配置页”，避免把运营内容再次拆散。

### 小程序前端改造

前端继续复用当前 `home` 和 `special-zone` 页面：

- `home/index.tsx`
  新增 `promotionZoneCover` 状态
- `loadSpecialZones()`
  新增请求 `promotion`
- 首页模板新增一个完整宽度的“活动专区”卡片/板块
- 点击行为跳转：
  `/pages/special-zone/index?type=promotion&title=优惠专区`

专区页改造固定为：

- `special-zone/index.tsx`
  支持 `type=promotion`
- 顶部 banner 数据改为从 `position=promotion` 读取
- 商品数据改为通过 `show_in_promotion_zone=true` 筛选
- 页面标题展示为“优惠专区”或路由传入标题

前端类型与服务同步扩展：

- `HomeBanner.position` union 增加 `promotion`
- `SpecialZoneCover.type` union 增加 `promotion`
- `Product` 类型增加 `show_in_promotion_zone`
- `productService.getHomeBanners()` 支持 `promotion`
- `specialZoneCoverService.getCovers()` 支持 `promotion`

## 接口与模型变化

### 后端新增或调整

- 商品接口新增查询参数：
  `show_in_promotion_zone`
- 首页轮播图接口支持：
  `position=promotion`
- 首页专区封面接口支持：
  `type=promotion`

### 前端类型新增或调整

- `Product.show_in_promotion_zone`
- `HomeBanner.position = 'promotion'`
- `SpecialZoneCover.type = 'promotion'`

### 商家后台表单项新增

- 商品编辑开关：`show_in_promotion_zone`
- 轮播图位置：`promotion`
- 专区封面类型：`promotion`

## 分阶段实施顺序

### Phase 1：平台底座与权限

- 新建 `stores` 域
- 新增 `Store`、成员、页面配置、支付配置、分账规则模型
- 历史数据迁移为主店
- 所有核心业务实体补 `store_id`
- 平台/店铺权限体系落地

### Phase 2：订单与支付重构

- 新增 `CheckoutOrder`、`SubOrder`
- 购物车按店铺分组
- 下单接口生成结算单和子单
- 支付成功后按“店铺 + 商品 SPU”拆单
- 子单驱动发货、售后、退款、分账记录

### Phase 3：登录改造

- 新增微信快捷登录接口
- 小程序接入手机号授权
- 用户资料初始化与默认昵称策略落地
- 登录弹层和动作守卫接入全站
- 商品分享落地页接入登录守卫与登录后回跳能力

### Phase 4：设计系统与核心页 UI 改版

- 重做全局 token 和组件层
- 先改首页、店铺、商品、购物车、下单、订单、我的页
- 把首页活动专区纳入首批核心运营板块
- 再统一其余页面风格

### Phase 5：分账闭环与后台增强

- 接入微信服务商分账流程
- 子单级分账记录、重试、异常处理
- 平台后台全局视图
- 店铺后台只看本店数据

## 测试与验收

### 后端测试

- `promotion` 相关 migration 成功，choices 和字段正常可用。
- 商品列表接口在 `show_in_promotion_zone=true` 时仅返回活动专区商品。
- `HomeBanner` 的 `promotion` 位置查询只返回活动专区轮播图。
- `SpecialZoneCover` 的 `promotion` 类型查询只返回活动专区封面。
- 商品导出、商品筛选和后台编辑回显正确包含 promotion 字段。

### 商家后台验收

- 能创建、编辑、删除活动专区封面。
- 能为活动专区配置轮播图。
- 能在商品上开启或关闭“优惠专区展示”。
- 不新增菜单的前提下，现有配置路径可完整完成活动专区配置。

### 小程序前端验收

- 首页在存在 promotion 封面时显示活动板块，不存在时隐藏。
- 点击活动板块能进入 `special-zone` 的 promotion 视图。
- 活动专区页能加载 promotion 轮播图和 promotion 商品。
- 活动专区无商品时，页面展示空态但不报错。
- 活动专区样式在未来 UI 重构中纳入首页首批改造范围。
- 用户通过商品分享链接首次打开小程序时，可以正常进入商品详情页。
- 该用户在未登录状态下点击“立即购买”时，会被正确引导到登录/注册/授权流程，而不是停留在无法下单的状态。
- 登录成功后会自动回到原商品详情页，并保留继续购买的路径。
- 已登录用户通过分享链接进入商品详情页时，不受额外阻断，可直接进入购买流程。

### 文档验收

- 文档文件创建成功。
- `docs/README.md` 能找到“平台升级实施计划 v3”入口。
- 文档可作为后续实施和拆任务的统一蓝图。

## 默认假设

- “活动板块”按当前系统语义等价于“第四种首页专区”，名称统一用 `promotion`，展示文案默认用“优惠专区”。
- 活动专区继续复用 `special-zone` 页面，不单独新增页面模板。
- 首页活动板块默认是一个独立的大卡片入口，不在首页直接渲染商品瀑布流。
- 活动专区商品仍由商品级开关控制，不单独维护手工商品列表。
- 文档保存路径默认使用 `docs/plan/2026-04-29-platform-upgrade-v3.md`。
- 商家后台第一阶段只做功能适配，不单独启动整套视觉重构。
- 分享商品链接默认落到商品详情页，登录完成后的回跳目标也以商品详情页为准。
